# 自动驾驶研发辅助 Copilot 设计文档

- 日期: 2026-04-13
- 状态: 已确认设计，待进入实施规划
- 目标级别: 可信版

## 1. 项目定位

### 1.1 对外定位

项目名称建议使用:

`自动驾驶研发辅助 Copilot`

副标题建议使用:

`基于 RAG + MCP 的多源技术知识检索与协同系统`

### 1.2 一句话简介

面向自动驾驶研发团队，接入算法文档、测试规范、数据处理说明、问题案例和版本资料，通过混合检索、引用式问答与 MCP 接口，提供研发知识检索、测试支持和问题排查能力。

### 1.3 为什么这样定位

当前项目的真实强项是:

- 多源文档接入与结构化处理
- BM25 + Dense Retrieval + RRF + Reranker 的混合检索链路
- 引用式回答与知识可追溯
- MCP 接口暴露与外部工具调用
- Query Trace、评估和 Dashboard 可观测能力

这些能力非常适合迁移为自动驾驶研发团队的知识与协同系统，但并不适合包装成感知模型训练、车端推理框架或工业级实时部署系统。因此本项目应定义为服务自动驾驶研发流程的辅助系统，而不是感知算法本身。

## 2. 目标与边界

### 2.1 目标

- 将自动驾驶研发资料统一接入同一知识底座
- 支持研发团队快速检索技术资料、测试规范和问题案例
- 提供引用式回答，降低技术问答中的幻觉风险
- 通过 MCP 暴露知识查询能力，支持 Agent 或工具调用
- 基于典型研发问答集建立可展示、可验证的评估链路

### 2.2 非目标

- 不包装为目标检测、车道线识别、BEV 等感知模型本身
- 不包装为训练平台、算力调度平台或实验管理平台
- 不包装为车端实时推理系统
- 不包装为算法量化、剪枝、编译优化框架

## 3. 目标用户

本项目主要服务:

- 自动驾驶算法工程师
- 测试 / 验证工程师
- 数据与工具链工程师
- 研发新成员和实习生

项目对外统一表述为:

`面向自动驾驶研发团队的知识与协同助手`

不将使用者限定为单一岗位，以避免项目看起来过度贴某个 JD。

## 4. 核心问题

自动驾驶研发团队普遍存在以下痛点:

- 算法说明、测试规范、问题案例和版本文档分散在多个位置
- 不同模块资料格式不一致，查找成本高
- 技术问答依赖人肉传承，新成员上手慢
- 测试和排障场景经常需要跨文档定位信息
- 现有资料难以被 Agent 或工具直接调用

本项目要解决的不是“替代工程师思考”，而是“缩短研发资料定位和问题排查链路”。

## 5. 可信版方案总览

可信版的原则是:

- 不做纯换皮包装
- 不重写核心底座
- 做真实的领域迁移

可信版应完成四类关键改造:

1. 自动驾驶领域数据替换
2. 自动驾驶研发问答样例补齐
3. 对外模块命名和文案迁移
4. 演示链路与评估样例补齐

优先级排序:

`领域数据 > 场景问答 > 演示链路 > 模块命名`

## 6. 技术架构

### 6.1 总体结构

可信版继续沿用现有项目的 6 层结构:

1. 知识资产层
2. 数据处理层
3. 检索层
4. 回答层
5. 协同接入层
6. 评估与观测层

### 6.2 知识资产层

建议接入以下五类资料:

- 算法 / 模块说明
- 测试规范与评测口径
- 数据处理与标注说明
- 问题案例与 FAQ
- 版本资料与变更记录

示例内容包括:

- 感知模块说明文档
- 数据格式和字段定义
- 测试脚本使用说明
- 评测指标解释
- 常见报错与排障步骤
- 版本迭代记录

### 6.3 数据处理层

继续使用现有 Ingestion Pipeline:

- Load
- Split
- Transform
- Embed
- Upsert

可信版的关键不在于重做 Pipeline，而在于补充自动驾驶研发语境下的元数据。建议新增或重点突出:

- `module`
- `task_type`
- `doc_type`
- `version`
- `source_team`

### 6.4 检索层

保留当前组合:

- BM25 负责术语、脚本名、指标名等精确匹配
- Dense Retrieval 负责语义相似召回
- RRF 负责多路结果融合
- Reranker 负责最终排序优化

这样可以适配自动驾驶研发中的混合型查询:

- 专业术语查询
- 场景问题查询
- 版本差异查询
- 测试与排障查询

### 6.5 回答层

继续采用引用式回答策略，输出内容应包含:

- 结论
- 来源文档
- 所属模块或版本
- 必要时给出“不足以回答”的明确提示

该层的目标不是生成泛泛建议，而是基于知识库给出可追溯回答。

### 6.6 协同接入层

继续使用 MCP Server 暴露知识能力，至少保留以下能力:

- 查询知识库
- 获取文档摘要
- 查看可用集合

项目对外可表述为:

`支持外部 Agent、IDE 辅助工具或内部研发工具直接调用知识库能力`

### 6.7 评估与观测层

保留现有 Dashboard、Trace 和评估机制，重点展示:

- Query Trace
- Retrieval Trace
- 典型问答评估结果
- 文档命中情况
- 回答引用完整性

此层是项目“可信”的重要支撑，因为它说明该系统不是简单问答演示，而是可观测、可验证的工程系统。

## 7. 自动驾驶领域语义映射

当前项目原本面向医疗知识场景，可信版不要求重命名所有内部代码模块，但需要完成对外语义映射。

### 7.1 对外命名建议

- Medical Demo -> Autonomous Driving Demo / AutoDrive Demo
- Guidelines -> 算法规范 / 测试规范
- Manuals -> 模块说明 / 工具说明
- Training -> 数据说明 / 培训资料 / 入门文档
- Quality Assistant -> 研发辅助 / 测试支持 / 问题排查助手

### 7.2 对外场景建议

主要讲三类场景:

- 研发知识检索
- 测试支持
- 问题排查

不把项目限定为单一的“感知部署”场景，以保持自然和通用性。

## 8. 可信版交付范围

### 8.1 必做项

- 重命名项目对外定位与简介
- 替换 demo 数据为自动驾驶研发资料
- 重写默认问答案例与演示脚本
- 增加自动驾驶场景的评估样例
- 重写 README、项目介绍和演示材料

### 8.2 可选增强项

- 按模块和版本优化 metadata 过滤体验
- 在 Dashboard 中加入更贴近研发场景的统计展示
- 增加 1 到 2 条 Agent 调用演示链路
- 为问题案例增加更明确的分类标签

## 9. 预计工期

基于现有底座做“可信版”迁移，合理工期为 `5-7 天`。

建议拆分为:

- 第 1 天: 确认项目定位、命名、数据目录和元数据策略
- 第 2-3 天: 整理并接入自动驾驶领域样例数据
- 第 4 天: 重写 README、项目介绍、查询样例和演示文案
- 第 5 天: 补齐评估样例、演示 runbook 和基础截图
- 第 6-7 天: 调整 Dashboard 文案与展示细节，完善简历与面试口径

## 10. 风险与控制

### 10.1 主要风险

- 只有文案迁移，没有真实领域数据支撑
- 场景过度聚焦某个岗位，看起来像定制包装
- 过度声称感知部署能力，被面试追问后失守
- 只有静态描述，没有 demo、评估和 trace 支撑

### 10.2 控制策略

- 以真实自动驾驶研发资料替换 demo 数据
- 使用“自动驾驶研发团队”而非单一岗位作为用户定义
- 明确项目属于研发支持层，不属于感知模型本身
- 保留并展示评估、Trace 和 Dashboard 能力

## 11. 简历与面试口径

### 11.1 简历标题

`自动驾驶研发辅助 Copilot`

### 11.2 简历一句话简介

面向自动驾驶研发团队设计并实现研发辅助 Copilot，基于 RAG + MCP 打通算法文档、测试规范、数据处理说明、问题案例和版本资料，支持多源知识接入、混合检索、引用式问答与 Agent 调用。

### 11.3 简历要点

- 设计并实现面向自动驾驶研发团队的多源知识检索系统，构建 `Load-Split-Transform-Embed-Upsert` 数据处理流水线，支持 PDF、Markdown、图文资料的统一接入与增量更新
- 搭建 `BM25 + Dense Retrieval + RRF + Reranker` 混合检索链路，提升研发文档、测试规范和问题案例场景下的召回与排序效果
- 实现带来源引用的问答与摘要能力，并通过 MCP 暴露查询接口，支持外部 Agent 或工具直接调用知识库能力
- 构建 Query Trace、Retrieval Trace 与评估面板，基于典型研发问答集验证系统效果，支撑可观测分析与演示展示

### 11.4 面试定义

推荐统一表述为:

`这个项目服务的是自动驾驶研发流程，而不是感知模型本身。我主要做的是把研发资料结构化接入、提升检索质量、提供引用式回答和工具化调用能力。`
